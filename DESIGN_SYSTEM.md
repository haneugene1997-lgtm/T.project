# SKT Legal Compliance Agent — Design System v1.0

> 정보·컴플라이언스 사전 검토 에이전트를 위한 UX/UI 설계 기준서
> Benchmark: **Google Gemini**, **Claude**
> 대상: `SKTLegalChat.jsx` (현 단일 페이지 React 앱)
> 작성일: 2026-04-21

---

## 0. 요약 (Executive Summary)

현재 에이전트는 **다크 글래스모피즘 + iOS 팔레트** 톤으로 완성도 있게 구현되어 있으나, 실제 업무 흐름(**문서 붙여넣기 → 검토 → 결과 해석 → BP 공유 판단**)을 빠르게 반복하기에는 몇 가지 병목이 있다.

- **진입 경험**: 처음 본 사용자는 "이게 단순 챗봇인지, 리포트 도구인지" 구분이 어려움. Gemini·Claude가 쓰는 **작업 카테고리(Intent) 기반 진입**을 도입하면 학습 곡선이 사라진다.
- **일상 업무 흐름**: 분석 결과가 풀스크린 오버레이로 열려 **대화와 분리**됨 → Claude의 Artifact 패턴처럼 **좌/우 분할 뷰**로 바꾸면 반복 검토가 30~40% 빨라진다.
- **신뢰감**: 리스크 점수·체크리스트의 시각 언어가 강하지만 **근거(law citation)가 hover·click depth에 숨겨짐** → Perplexity/Claude식 인라인 출처 배지가 필요.
- **BP 외부 공유**: 현재 플로우에 없는 독립 유즈케이스. "공유 전 사전 점검" **Pre-Share Gate** 패턴을 신규 제안.

본 문서는 **디자인 토큰 → 컴포넌트 → 패턴 → 로드맵** 순으로 설계 기준을 제공한다. 개발자는 이 문서를 기반으로 단계적 리팩터를 진행할 수 있다.

## 1. 디자인 원칙 (Design Principles)

| # | 원칙 | 의미 | 판단 기준 |
|---|------|------|-----------|
| 1 | **Trust by Default** | 모든 판단에는 **근거가 보여야** 한다 | 리스크 표시 옆에 법령·조항 클릭 가능 배지 |
| 2 | **Glanceable Risk** | 한 눈에 위험도를 읽을 수 있어야 한다 | 색·수치·형상의 **삼중 중복** (색각 이상 대응) |
| 3 | **Flow over Form** | 폼이 아닌 **대화**로 입력받고, 필요한 곳만 구조화한다 | 붙여넣기 1회 = 검토 1회 |
| 4 | **Reversible Actions** | 모든 파괴적 액션은 **되돌릴 수 있어야** 한다 | 히스토리 삭제·파일 제거·탭 전환 |
| 5 | **Progressive Disclosure** | 복잡한 결과는 **필요할 때만** 펼친다 | 카드 요약 → 이슈 목록 → 상세 펼침 |
| 6 | **Compliance as Guardrail** | 에이전트가 **금지선을 넘지 않게** 한다 | 사전 동의·검토 권고·PII 감지 배너 |

## 2. 현재 디자인 감사 (Audit)

### 2.1 요약 점수

| 영역 | 점수 | 비고 |
|------|------|------|
| 토큰 체계 | 4/10 | 인라인 스타일에 하드코딩된 헥스·rgba 반복 |
| 컴포넌트 완결성 | 6/10 | 기본 상태만 있고 hover/focus/disabled/loading 누락 다수 |
| 네이밍 일관성 | 7/10 | `RC`, `VT`, `glass()` 등 축약명 — 확장 시 가독성 저하 |
| 접근성 | 3/10 | ARIA/키보드/색대비 대부분 미검증 |
| 온보딩 | 5/10 | 인사말 + 빠른 질문 4개 — 의도·결과물 예시 부재 |
| 일상 흐름 | 6/10 | 대화 플로우는 매끄럽지만 결과↔대화 왕복이 무거움 |
| **종합** | **5.2/10** | 실제 업무 시 마찰 발생 구간 확인됨 |

### 2.2 주요 일관성 이슈

| 이슈 | 위치 | 권고 표준 |
|------|------|-----------|
| 색상 하드코딩 (`#007aff`, `rgba(255,255,255,0.04)` 등) 100+회 | 컴포넌트 전역 | `--color-accent-primary` 토큰화 |
| 간격 값 임의(`4, 6, 7, 8, 10, 12, 14, 16, 18, 20, 24, 28...`) | padding/margin 전역 | **4pt 그리드**로 통일: 4 · 8 · 12 · 16 · 24 · 32 · 48 |
| 폰트 크기 임의(`9, 10, 11, 12, 13, 14, 15, 20, 48...`) | 텍스트 전역 | 8단계 **타입 스케일**로 축소 |
| Radius 혼재(`4, 5, 6, 7, 8, 10, 11, 12, 14, 16, 18`) | 카드/버튼/칩 | 4단계로 축소: `xs 4 · sm 8 · md 12 · lg 16 · xl 24` |
| 이모지 아이콘만 사용 (⚖️, 📎, 📊 …) | 브랜드·법령·액션 전역 | 브랜드 요소는 **SVG 아이콘**으로 교체, 법령 이모지는 유지 가능 |
| `glass()` helper가 **스타일 객체 반환** — 다크 전용 | Radar/카드 | Light 모드 대비 확장 어려움 (토큰화 필요) |
| 리스크 레벨 라벨과 색 **중복 정의** | `RC`와 `RadarChart` | 단일 토큰화 |

## 3. 벤치마크 (Gemini & Claude)

### 3.1 Gemini에서 가져올 패턴

| 패턴 | 설명 | 적용 포인트 |
|------|------|-------------|
| **Intent 기반 홈** | "Write · Plan · Learn · Code" 카테고리 버튼 → 빠른 시작 | 홈에 "빠른 질의 · 문서 검토 · BP 공유 점검 · 용어 설명" 4개 인텐트 카드 |
| **Suggestion Chips (동적)** | 대화 후에도 후속 질문을 칩으로 제시 | 이슈 카드 아래 "관련 조항 더 보기", "수정 문구 초안", "내부 공유 가능 여부" |
| **그라디언트 포인트** | 입력 포커스·로딩 시 은은한 그라디언트 이동 | 전송 버튼·로딩 스피너·리스크 배지 하이라이트 |
| **Gem (Custom Agent)** | 자주 쓰는 페르소나(계약·약관·PIA)를 저장 | 상단 탭: `전체`·`계약검토`·`약관검토`·`PIA·영향평가` |
| **큰 프롬프트 박스** | 홈의 입력이 **시각적 주인공** | 빈 상태에서 입력 영역을 80% 크기로 중앙에 배치 |

### 3.2 Claude에서 가져올 패턴

| 패턴 | 설명 | 적용 포인트 |
|------|------|-------------|
| **Artifact 분할 뷰** | 대화 좌 + 산출물 우, 동시 확인 가능 | 현재 풀스크린 오버레이 → **우측 패널**로 이동 |
| **Project** | 관련 문서·대화를 프로젝트 단위로 묶음 | "2026-Q2 OO서비스 약관 검토" 프로젝트 폴더 |
| **명확한 Sender 라벨** | 시스템·에이전트·사용자를 명확히 구분 | 현재 좋은 편 — 유지, 다만 아바타 대비 개선 |
| **Inline Citation** | 주장 옆에 `[1]`, 클릭 시 원본 이동 | 이슈 설명 내 조항명을 **인라인 배지**로 |
| **Thinking 상태** | "조항 DB 확인 중…" 같은 **투명한 진행 상황** | 단순 도트 → "문서 구조 파싱 → 법령 매칭 → 리스크 집계" 3단계 표시 |
| **Copy·Share·Pin** | 모든 메시지·산출물에 공통 액션 | 메시지 hover 시 우측 상단 액션 도크 |

### 3.3 SKT 컨텍스트에서 절대 가져오면 안 되는 것

- **외부 이미지 생성 / 웹서치 인라인** — 폐쇄 내부망 컴플라이언스와 충돌
- **과도한 감정 이모지** — 법무 맥락에선 경량화 필요
- **인터넷에 자동 업로드되는 공유 링크** — BP 공유 사전점검 취지와 정면 배치

## 5. 컴포넌트 명세 (Component Specs)

| Variant | 예시 | 사용 시 |

#### States

| State | 시각 | 동작 |
|-------|------|------|
| default | 기본 배경 | — |
| hover | elev-1 → elev-2 | cursor: pointer |
| focus-visible | 2px 포커스 링 `var(--brand-500)` | 키보드만 표시 |
| active | 아주 약한 scale(0.98) | pressed 느낌 |
| loading | 텍스트 숨기고 스피너 | 클릭 비활성 |
| disabled | opacity 0.5, cursor not-allowed | — |

#### Do / Don't

| ✅ Do | ❌ Don't |
|------|---------|
| primary는 화면당 1개 | 3개 이상의 primary 버튼 |
| 아이콘 버튼엔 `aria-label` | 아이콘만 두고 레이블 없음 |
| 위험 액션엔 confirm 다이얼로그 | 즉시 실행되는 "전체 삭제" |

### 5.2 Composer (입력 영역)

**현재**: textarea + 전송 버튼 + 파일 첨부 버튼.
**권고**: 입력 영역을 **Composer** 컴포넌트로 승격.

#### 하위 슬롯

| 슬롯 | 내용 |
|------|------|
| `intentChips` | (옵션) 상단에 "빠른 질의 · 문서 검토 · BP 공유 점검" 토글 |
| `attachmentStrip` | 첨부된 파일 썸네일 목록 (복수 지원) |
| `textarea` | 자동 확장, max-height 160px, 1~8줄 |
| `toolbar` | 좌: `+` 첨부 / 우: `↑` 전송 · `⇧⏎` 안내 |
| `hint` | 하단 캡션 (규정 고지 + 단축키) |

#### Props

| Prop | Type | Default | 설명 |
|------|------|---------|------|
| value | string | — | 바인딩 |
| onSubmit | `(payload) => void` | — | — |
| onAttach | `(File[]) => void` | — | 여러 파일 지원 |
| mode | `quick \| analyze \| preshare` | `quick` | 인텐트 모드 |
| disabled | boolean | false | — |
| maxFiles | number | 5 | — |

#### 상태

| 상태 | 동작 |
|------|------|
| empty | placeholder 표시 |
| typing | 전송 버튼 그라디언트 활성 |
| hasAttachment | 첨부 스트립 펼침 |
| pasting-large-text | 5,000자 이상 붙여넣기 감지 → **"문서 검토 모드로 전환?" 토스트** |
| blocked | `sensitive-pattern` 감지 (주민번호·카드번호 정규식) → 인라인 경고 + 마스킹 제안 |

> ⚠️ **Sensitive Pattern 감지**는 이 에이전트의 핵심 신뢰 장치. `\d{6}-[1-4]\d{6}` (주민번호), `\d{4}-?\d{4}-?\d{4}-?\d{4}` (카드번호), 이메일, 전화번호 등이 입력 중 발견되면 **전송 전 경고 배너**를 보여주고, `자동 마스킹 적용`·`그대로 전송`·`취소` 3개 액션을 제공한다.

#### 키보드

| 키 | 동작 |
|----|------|
| `Enter` | 전송 |
| `Shift + Enter` | 줄바꿈 |
| `Cmd/Ctrl + K` | 명령 팔레트 |
| `Cmd/Ctrl + Enter` | 전송 (일부 유저 선호) |
| `Cmd/Ctrl + U` | 파일 첨부 |
| `Esc` | 첨부 취소 또는 분석 패널 닫기 |

## 6. 패턴 라이브러리

### 6.1 패턴: Onboarding (첫 진입)

#### 목표
"이 에이전트가 무엇을 해주는지, 30초 안에 체감한다."

#### 흐름 (Gemini식 Intent + Claude식 샘플)

```
STEP 1. Hero
  "문서나 질문을 보내세요. 컴플라이언스 리스크를 사전 검토합니다."
  [큰 Composer]

STEP 2. Intent Cards (4개)
  ┌─────────┬─────────┬─────────┬─────────┐
  │💬 빠른질의│📄 문서검토│🛡️ BP 공유 │📚 조항설명│
  │조항·용어 │계약·약관 │공유전 점검│판례·해석 │
  │질문 1문장│ 파일 업로드│ 자동 진단 │ 자연어   │
  └─────────┴─────────┴─────────┴─────────┘

STEP 3. "시연 보기" (옵션)
  "샘플 계약서로 30초 분석 시연 보기"
  → 미리 준비된 계약서로 자동 분석 플레이

STEP 4. Law DB (하단, 접힘)
  커버하는 6개 법령 · 최신 업데이트 날짜
```

#### 체크포인트

- 첫 2분 내 첫 분석 결과 도달률을 성공 지표로.
- Intent 카드 클릭 시 Composer의 mode만 바뀌는 **무파편 전환** (페이지 이동 X).
- 사용자가 빈 상태에서 `Enter`를 눌러 입력 없이 제출해도 **에러가 아닌 빈 상태 안내** 반환.

### 6.2 패턴: Daily Work Flow (붙여넣기 → 검토 → 해석 → 공유 판단)

#### 목표
반복 검토 업무의 **클릭 수를 절반으로**.

#### 현재 vs 개선 (핵심 3가지)

| 단계 | 현재 | 개선 |
|------|------|------|
| 결과 확인 | 풀스크린 오버레이 (대화 숨김) | **우측 Drawer**: 대화와 분석 동시 확인 |
| 탭 이동 | 대시보드 ↔ 이슈 ↔ 체크리스트 (풀페이지 스위치) | 좌측 **고정 사이드 네비**로 즉시 점프 |
| 반복 검토 | 새 질문마다 컨텍스트 설명 필요 | **"이 검토 결과 기반으로 질문"** 버튼 → 동일 분석 세션 유지 |

#### Split-Pane 레이아웃 (데스크톱 ≥ 1024px)

```
┌─────────────────────────────────────────────────────────────┐
│  Header                                                      │
├────────────────────────────────┬─────────────────────────────┤
│                                │ 📊 분석 리포트              │
│  대화                          │ ┌─ Tabs ─────────────────┐ │
│  [user bubble]                 │ │ 대시보드 이슈 체크리스트│ │
│  [agent analysis card]         │ └─────────────────────────┘ │
│  [suggestion chips]            │                              │
│  ...                           │ (본문)                       │
│                                │                              │
├────────────────────────────────┴─────────────────────────────┤
│  Composer (전역 고정)                                         │
└─────────────────────────────────────────────────────────────┘
```

- 분석 결과 카드의 "상세 리포트 보기" 클릭 → **우측 drawer 열림** (좌측 대화 유지).
- drawer 상단에 "닫기 / 팝아웃(새 창) / PDF / JSON" 공통 액션.
- 사용자가 드래그로 분할 폭 조절 가능, 로컬에 저장.

#### Mobile 대응 (< 768px)

- drawer는 **하단에서 올라오는 bottom sheet** (70~95% 높이).
- 스와이프 다운으로 접기/펼치기 3단계 (peek · half · full).

### 6.3 패턴: Contextual Re-Ask (검토 결과 기반 재질의)

각 이슈 카드 하단의 Suggestion Chips 중 "수정 초안 주세요" 등을 누르면:

1. 해당 이슈 ID를 자동 첨부한 시스템 메시지가 생성됨.
2. 분석 세션은 유지 — 새 질문이 같은 문맥(session_id)로 들어감.
3. 답변은 대화 쪽에 노출되고, drawer의 해당 이슈는 **"수정안 있음 • 새 탭으로 보기"** 뱃지가 추가됨.

### 6.4 패턴: BP 외부 공유 사전 점검 (Pre-Share Gate)

#### 트리거
- 인텐트 카드 "🛡️ BP 공유 점검" 선택
- 분석 결과 drawer의 하단 액션 "외부 공유 검토"

#### 흐름

```
1. 수신 선택
   (드롭다운: 기존 BP 목록 + 신규 추가)
2. 공유 목적 입력 (자유 텍스트, 100자)
3. 원문 첨부 또는 기존 분석 세션 선택
4. 자동 점검 실행
   - PII/민감정보 감지
   - 내부 코드명(프로젝트 네이밍) 감지
   - 위수탁 계약 체결 여부(사용자가 보유한 계약 리스트와 매칭)
   - 영업비밀 키워드 감지
5. 결과 카드
   - ✅ 통과 항목 접기
   - ❌/⚠️ 항목 펼쳐서 하나씩 "마스킹" / "제거" / "무시 사유 입력" 액션
6. 산출
   - "공유 가능 버전" docx/pdf 다운로드 + 점검 이력 PDF
```

#### 디자인 포인트
- 색 코드는 리스크 레벨과 동일하지만 **아이콘은 🛡️ 계열**로 구분.
- 결과 PDF에는 점검 일시·점검자·버전 해시를 포함해 **감사 가능**하게.

### 6.5 패턴: Empty / Error / Loading

| 시나리오 | 현재 | 개선 |
|----------|------|------|
| 빈 대화 | 좋음 (Hero + 빠른 질문) | Intent 카드 추가 |
| 업로드 실패 (사이즈/형식) | silent | 인라인 토스트 + "다른 형식 시도" |
| API 429 / 네트워크 | `⚠️ 오류` 단문 | `재시도` 버튼, 지수백오프 |
| 빈 분석 결과 | 노출 안 됨 | "리스크가 식별되지 않았습니다 — 수동 확인 체크리스트 보기" |
| 히스토리 비어있음 | 아이콘 안 보임 | 첫 진입 시 "히스토리 없음 · 첫 분석을 시작해보세요" |

## 8. 접근성 (A11y)

### 8.1 키보드

| 요소 | 키보드 |
|------|--------|
| Composer | `Enter` 전송 / `Shift+Enter` 줄바꿈 |
| 분석 탭 | `←/→` 이동, `Enter` 선택 |
| 이슈 카드 | `Space/Enter` 펼침·접힘 |
| Drawer | `Esc` 닫기, `F6` 포커스 이동 |
| 전역 | `Cmd/Ctrl+K` 명령 팔레트 |

### 8.2 스크린리더

- `<main>` / `<aside>` / `<nav>` / `<header>` 랜드마크 필수.
- 분석 카드는 `<article aria-labelledby=…>`.
- 진행 상태는 `aria-live="polite"` 영역에 출력.
- Score Gauge에 `role="img"` + `aria-label` (숫자 말하기).

### 8.3 색·모션

- 모든 상태는 **색만으로 구분되지 않게** 형상 병행.
- `prefers-reduced-motion: reduce` 시 카운트업·라디안 애니메이션 제거.
- 포커스 링은 2px, 배경과 3:1 이상 대비.

## 10. 자주 하는 결정 가이드

| 상황 | 선택 |
|------|------|
| 버튼이 3개 이상 필요해 보임 | **가장 중요한 것만 primary**, 나머지 secondary/ghost |
| 새로운 색이 필요해 보임 | **먼저 기존 토큰에 매핑** 시도, 그래도 안 되면 시맨틱/브랜드 토큰 추가 논의 |
| "추가 정보"가 필요해 보임 | 툴팁 < 펼침 카드 < drawer — **단계적 공개** |
| 아이콘을 혼자 쓸 수 있을까? | `aria-label` 달고 hover tooltip 붙이고 나서 YES |
| 오류 메시지 톤 | "무엇이 · 왜 · 다음에 할 일" 3요소 포함, 감정적 표현 ❌ |
